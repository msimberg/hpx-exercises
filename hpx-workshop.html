Task-based programming

Task-based programming?

  • A programming model for parallel programs
  • Not using (POSIX) threads explicitly
  • Not using mutexes etc. to synchronize work
  • Creating an directed acyclic graph (DAG) of tasks
    • Nodes are tasks
    • Edges are dependencies
  • Implicit: not fork-join

Task-based programming: benefits

  • Simpler
    • No mutexes etc.
    • Data dependencies control the flow of your program

  • Better utilization with small tasks

  • Task-based programming gets better the more you use it

  • No fork-join

Standard C++ task-based programs: serializing tasks with async

future<void> f = async(&a);
future<void> g = async(
    [f = move(f)]() mutable { f.wait(); b(); });
future<void> h = async(
    [g = move(g)]() mutable { g.wait(); c(); });
async([h = move(h)]() mutable { h.wait(); d(); });

Standard C++ task-based programs: fan out with shared_future<T> and async

shared_future<void> f = async(&a);
// shared_future<void> f = async(&a).share();

async([f]() { f.wait(); d(); });

Standard C++ task-based programs: fan in with async

auto f = async(&a);
auto g = async(&b);
auto h = async(&c);
async([f = move(f), g = move(g), h = move(h)]()
    { f.wait(); g.wait(); h.wait(); d(); });

HPX

HPX

HPX: High Performance paralleX

  • An idea (execution model) and an implementation
  • Single-node HPX = High performance standard library
  • Developed by the Ste||ar Group

HPX: Idea

Avoid and embrace:

  • Starvation (have enough work)
  • Latency (hide latency)
  • Overheads (lightweight threads)
  • Waiting (don't wait for shared resources)

HPX: Implementation

  • Standards conformant C++ library
  • Lightweight threads run on top of OS-threads
    • Cooperative multi-tasking
  • Many usability improvements
    • combinators for constructing DAGs
    • executors
  • (AGAS)
  • (local & remote execution)
  • (…)

HPX: Implementation

  • OS-thread pools manage tasks (a.k.a. lightweight threads or HPX threads)
  • One OS-thread per core (typically)
  • One scheduling loop per OS-thread
  • hpx::async (and others) add tasks to scheduler queues
  • One task at a time runs in on an OS-thread
  • Schedulers can steal from each other's queues

HPX: Why not OpenMP, TBB, …?

  • No standards conformance
  • Fork-join (often) imposed
  • Nested parallelism difficult or impossible
  • Dynamic DAGs difficult or impossible
  • No standards conformance

HPX task-based programs: execution policies

  • Standard execution policies
    • seq: sequentially
    • par: in parallel
    • par_unseq: in parallel + SIMD
  • Additional execution policies in HPX
    • seq(task): sequentially, return a future
    • par(task): in parallel, return a future
    • par_unseq(task): in parallel + SIMD, return a future

Really writing task-based programs: ground rules

  • Take inputs and return outputs
  • Make the tasks small enough, but not too small (min. 1 ms)
    • Make the grain-size adjustable
  • Don't
    • Add unnecessary dependencies
    • Share state unnecessarily
    • Use mutexes etc.
  • Can yield/suspend implicitly or explicitly (cooperative multi-tasking)
    • Implicitly e.g. on get if future is not ready
    • Explicitly if you need to let other threads run (unusual)
    • Never block the underlying OS-thread

// In main thread
std::vector<std::thread> threads;

  threads.push_back(std::thread(&scheduling_loop));

void scheduling_loop() {
  while (task = tasks.pop()) {
